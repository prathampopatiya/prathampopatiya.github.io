<h1>Kernel Programming Basics</h1>
<h2>Differences Between User Mode and Kernel Mode Development</h2>
<h2>| Specification | User Mode | Kernel Mode |
|---|---|---|
| Unhandled exceptions | Crashes the process | Crashes the system |
| Terminations | When a process terminates, all private memory and resources are freed automatically | If a driver unloads without freeing everything it was using, there is a leak, only resolved in the next boot |
| IRQL | <code>PASSIVE_LEVEL (0)</code> | <code>DISPATCH_LEVEL (2)</code> or higher |
| Return values | API errors are sometimes ignored | Should almost never ignore errors |
| Bad Coding | Typically localized to the process | Can affect the entire system |
| Testing and Debugging | Typical testing and debugging done on the developer’s machine | Debugging must usually be done with another machine |
| Libraries | Can use almost any C/C++ library (e.g. STL, Boost) | Most standard libraries cannot be used |
| Exception Handling | Can use C++ exceptions or Structured Exception Handling (SEH) | Only SEH can be used |
| C++ Usage | Full C++ runtime available | No C++ runtime |</h2>
<h1>The Kernel API</h1>
<p>Kernel drivers use exported functions from kernel components. These functions are referred to as the <strong>Kernel API</strong>. Most functions are implemented within the kernel module itself (<code>ntoskrnl.exe</code>), but some may be implemented by other kernel modules such as the HAL (<code>hal.dll</code>).</p>
<p>| Prefix  | Meaning                            | Example                      |
| ------- | ---------------------------------- | ---------------------------- |
| <code>Ex</code>    | General executive functions        | <code>ExAllocatePoolWithTag</code>      |
| <code>Ke</code>    | General kernel functions           | <code>KeAcquireSpinLock</code>          |
| <code>Mm</code>    | Memory manager                     | <code>MmProbeAndLockPages</code>        |
| <code>Rtl</code>   | General runtime library            | <code>RtlInitUnicodeString</code>       |
| <code>FsRtl</code> | File system RTL                    | <code>FsRtlGetFileSize</code>           |
| <code>Flt</code>   | File system mini-filter            | <code>FltCreateFile</code>              |
| <code>Ob</code>    | Object manager                     | <code>ObReferenceObject</code>          |
| <code>Io</code>    | I/O manager                        | <code>IoCompleteRequest</code>          |
| <code>Se</code>    | Security                           | <code>SeAccessCheck</code>              |
| <code>Ps</code>    | Process manager                    | <code>PsLookupProcessByProcessId</code> |
| <code>Po</code>    | Power manager                      | <code>PoSetSystemState</code>           |
| <code>Wmi</code>   | Windows Management Instrumentation | <code>WmiTraceMessage</code>            |
| <code>Zw</code>    | Native API wrappers                | <code>ZwCreateFile</code>               |
| <code>Hal</code>   | Hardware abstraction layer         | <code>HalExamineMBR</code>              |
| <code>Cm</code>    | Config manager                     | <code>CmRegisterCallbackEx</code>       |</p>
<blockquote>
<p><strong>Note</strong></p>
<p>If you look at the exported functions list from <code>ntoskrnl.exe</code>, you’ll find many functions that are not documented in the Windows Driver Kit.</p>
</blockquote>
<hr>
<h1>Dynamic Memory Allocation</h1>
<p>Drivers often need to allocate memory dynamically. As discussed in Chapter 1, kernel thread stack size is rather small, so any large chunk of memory should be allocated dynamically.</p>
<p>The kernel provides two general memory pools for drivers to use:</p>
<ul>
<li><strong>Paged Pool</strong> — Memory pool that can be paged out if required.</li>
<li><strong>Non-Paged Pool</strong> — Memory pool that is never paged out and is guaranteed to remain in RAM.</li>
</ul>
<blockquote>
<p><strong>Note</strong></p>
<p>The non-paged pool is a “better” memory pool because it can never incur a page fault.</p>
</blockquote>
<p>Drivers should use this pool sparingly and only when necessary. In most other cases, drivers should use the paged pool.</p>
<p>The <code>POOL_TYPE</code> enumeration represents the pool types. Although the enumeration contains many pool variants, drivers commonly use only:</p>
<ul>
<li><code>PagedPool</code></li>
<li><code>NonPagedPool</code></li>
<li><code>NonPagedPoolNx</code> (non-paged pool without execute permissions)</li>
</ul>
<p>| Function                     | Description                                                          |
| ---------------------------- | -------------------------------------------------------------------- |
| <code>ExAllocatePool</code>             | Allocates memory from one of the pools with a default tag. Obsolete. |
| <code>ExAllocatePoolWithTag</code>      | Allocates memory from one of the pools with a specified tag          |
| <code>ExAllocatePoolZero</code>         | Same as <code>ExAllocatePoolWithTag</code>, but zeroes the allocated memory     |
| <code>ExAllocatePoolWithQuotaTag</code> | Allocates memory and charges the current process quota               |
| <code>ExFreePool</code>                 | Frees allocated memory                                               |</p>
<blockquote>
<p><strong>Tip</strong></p>
<p><code>ExAllocatePool</code> internally calls <code>ExAllocatePoolWithTag</code> using the tag <code>enoN</code> (<code>none</code> reversed).</p>
<p>Older Windows versions used <code>mdW</code> (<code>WDM</code> reversed).</p>
<p>Prefer using <code>ExAllocatePoolWithTag</code> instead.</p>
</blockquote>
<h2>Sample Code</h2>
<pre><code class="language-cpp">#include &#x3C;ntddk.h>
#include &#x3C;ntstatus.h>
#include &#x3C;wdm.h>

#define DRIVER_TAG 'dcba'

void SampleUnload(_In_ PDRIVER_OBJECT DriverObject) {
    UNREFERENCED_PARAMETER(DriverObject);
    DbgPrint(("Sample Driver Unload called\n"));
}

UNICODE_STRING g_RegistryPath;

extern "C"
NTSTATUS DriverEntry(PDRIVER_OBJECT DriverObject, PUNICODE_STRING RegistryPath) {
    UNREFERENCED_PARAMETER(RegistryPath);

    DriverObject->DriverUnload = SampleUnload;

    DbgPrint(("Sample driver initialized successfully\n"));

    g_RegistryPath.Buffer = (WCHAR*)ExAllocatePool2(
        PagedPool,
        RegistryPath->Length,
        DRIVER_TAG
    );

    if (g_RegistryPath.Buffer == nullptr) {
        DbgPrint(("Failed to allocate memory\n"));
        return STATUS_INSUFFICIENT_RESOURCES;
    }

    g_RegistryPath.MaximumLength = RegistryPath->Length;

    RtlCopyUnicodeString(
        &#x26;g_RegistryPath,
        (PUNICODE_STRING)RegistryPath
    );

    DbgPrint(("Original registry path: %wZ\n"), RegistryPath);
    DbgPrint(("Copied registry path: %wZ\n"), &#x26;g_RegistryPath);

    OSVERSIONINFOW osv = {0};
    osv.dwOSVersionInfoSize = sizeof(OSVERSIONINFOW);

    if (NT_SUCCESS(RtlGetVersion(&#x26;osv))) {
        DbgPrint(("OS Major Version: %lu\n"), osv.dwMajorVersion);
        DbgPrint(("OS Minor Version: %lu\n"), osv.dwMinorVersion);
        DbgPrint(("OS Build Number: %lu\n"), osv.dwBuildNumber);
    }

    return STATUS_SUCCESS;
}
</code></pre>

<h1>Linked Lists</h1>
<p>The Windows kernel uses circular doubly linked lists in many of its internal data structures.</p>
<p>For example, all processes on the system are managed by <code>EPROCESS</code> structures connected in a circular doubly linked list. The head of this list is stored in the kernel variable <code>PsActiveProcessHead</code>.</p>
<pre><code class="language-cpp">typedef struct _LIST_ENTRY {
    struct _LIST_ENTRY* Flink;
    struct _LIST_ENTRY* Blink;
} LIST_ENTRY, *PLIST_ENTRY;
</code></pre>
<p>A <code>LIST_ENTRY</code> is embedded inside another structure.</p>
<p>For example, the <code>EPROCESS</code> structure contains a member named <code>ActiveProcessLinks</code> of type <code>LIST_ENTRY</code>.</p>
<p>To retrieve the parent structure from a <code>LIST_ENTRY</code>, the kernel uses the <code>CONTAINING_RECORD</code> macro.</p>
<h2>Common Linked List Functions</h2>
<p>| Function                      | Description                                     |
| ----------------------------- | ----------------------------------------------- |
| <code>InitializeListHead</code>          | Initializes an empty list                       |
| <code>InsertHeadList</code>              | Inserts an item at the head                     |
| <code>InsertTailList</code>              | Inserts an item at the tail                     |
| <code>IsListEmpty</code>                 | Checks whether the list is empty                |
| <code>RemoveHeadList</code>              | Removes the head item                           |
| <code>RemoveTailList</code>              | Removes the tail item                           |
| <code>RemoveEntryList</code>             | Removes a specific item                         |
| <code>ExInterlockedInsertHeadList</code> | Atomically inserts at the head using a spinlock |
| <code>ExInterlockedInsertTailList</code> | Atomically inserts at the tail using a spinlock |
| <code>ExInterlockedRemoveHeadList</code> | Atomically removes the head item                |</p>

<h1>The Driver Object</h1>
<pre><code class="language-cpp">NTSTATUS DriverEntry(
    PDRIVER_OBJECT DriverObject,
    PUNICODE_STRING RegistryPath
)
</code></pre>
<p>The <code>DriverEntry</code> routine accepts two parameters:</p>
<ol>
<li><code>DriverObject</code></li>
<li><code>RegistryPath</code></li>
</ol>
<p>The driver uses the <code>DriverObject</code> structure to initialize supported driver operations.</p>
<p>One important part of <code>DRIVER_OBJECT</code> is the <code>MajorFunction</code> array.</p>
<p>This array stores dispatch routines for different I/O request types.</p>
<h2>Major Functions</h2>
<p>| Major Function                   | Description                                           |
| -------------------------------- | ----------------------------------------------------- |
| <code>IRP_MJ_CREATE</code>                  | Create operation (<code>CreateFile</code>, <code>ZwCreateFile</code>)       |
| <code>IRP_MJ_CLOSE</code>                   | Close operation (<code>CloseHandle</code>, <code>ZwClose</code>)            |
| <code>IRP_MJ_READ</code>                    | Read operation (<code>ReadFile</code>, <code>ZwReadFile</code>)             |
| <code>IRP_MJ_WRITE</code>                   | Write operation (<code>WriteFile</code>, <code>ZwWriteFile</code>)          |
| <code>IRP_MJ_DEVICE_CONTROL</code>          | Invoked through <code>DeviceIoControl</code>                     |
| <code>IRP_MJ_INTERNAL_DEVICE_CONTROL</code> | Similar to device control but for kernel-mode callers |
| <code>IRP_MJ_SHUTDOWN</code>                | Invoked during system shutdown                        |
| <code>IRP_MJ_CLEANUP</code>                 | Invoked when the last handle to a file object closes  |
| <code>IRP_MJ_PNP</code>                     | Plug and Play callback                                |
| <code>IRP_MJ_POWER</code>                   | Power manager callback                                |</p>

<h1>Object Attributes</h1>
<p>A common structure used by many kernel APIs is <code>OBJECT_ATTRIBUTES</code>.</p>
<pre><code class="language-cpp">typedef struct _OBJECT_ATTRIBUTES {
    ULONG Length;
    HANDLE RootDirectory;
    PUNICODE_STRING ObjectName;
    ULONG Attributes;
    PVOID SecurityDescriptor;
    PVOID SecurityQualityOfService;
} OBJECT_ATTRIBUTES;

typedef OBJECT_ATTRIBUTES* POBJECT_ATTRIBUTES;
typedef const OBJECT_ATTRIBUTES* PCOBJECT_ATTRIBUTES;
</code></pre>
<p>The structure is typically initialized using the <code>InitializeObjectAttributes</code> macro.</p>
<h2>Object Attribute Flags</h2>
<p>| Flag                                | Description                                              |
| ----------------------------------- | -------------------------------------------------------- |
| <code>OBJ_INHERIT</code>                       | Returned handle is inheritable                           |
| <code>OBJ_PERMANENT</code>                     | Object remains alive even after all handles are closed   |
| <code>OBJ_EXCLUSIVE</code>                     | Object allows exclusive access                           |
| <code>OBJ_CASE_INSENSITIVE</code>              | Performs case-insensitive lookup                         |
| <code>OBJ_OPENIF</code>                        | Opens existing object if it exists                       |
| <code>OBJ_OPENLINK</code>                      | Opens the symbolic link itself instead of resolving it   |
| <code>OBJ_KERNEL_HANDLE</code>                 | Creates a kernel-only handle                             |
| <code>OBJ_FORCE_ACCESS_CHECK</code>            | Forces access checks even in kernel mode                 |
| <code>OBJ_IGNORE_IMPERSONATED_DEVICEMAP</code> | Uses process device map instead of impersonated user map |
| <code>OBJ_DONT_REPARSE</code>                  | Prevents following reparse points                        |</p>
