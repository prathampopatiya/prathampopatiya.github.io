19:Tf73,<h1>Reverse Engineering a Simple Binary</h1>
<p>Reverse engineering is the process of discovering the principles, structure, and functioning of a system through analysis of its artifacts. In this post, we'll walk through reversing a simple C binary using common tools and techniques.</p>
<h2>Tools You'll Need</h2>
<p>Before we begin, make sure you have these tools installed:</p>
<pre><code class="language-bash"># Install dependencies on Linux
sudo apt-get install binutils gdb radare2

# Or using homebrew on macOS
brew install binutils gdb radare2
</code></pre>
<h2>The Binary</h2>
<p>Let's start with this simple C program:</p>
<pre><code class="language-c">#include &#x3C;stdio.h>
#include &#x3C;string.h>

int verify_password(char *input) {
    char password[] = "superSecret123";
    
    if (strcmp(input, password) == 0) {
        return 1;
    }
    return 0;
}

int main() {
    char buffer[50];
    
    printf("Enter password: ");
    scanf("%49s", buffer);
    
    if (verify_password(buffer)) {
        printf("Access granted!\\n");
    } else {
        printf("Access denied!\\n");
    }
    
    return 0;
}
</code></pre>
<h2>Compilation and Initial Analysis</h2>
<p>First, let's compile the binary with debugging symbols:</p>
<pre><code class="language-bash">gcc -g -o verify_bin verify.c
</code></pre>
<p>Now let's examine it with <code>strings</code> to see what we can learn:</p>
<pre><code class="language-bash">$ strings verify_bin | grep -i secret
superSecret123
</code></pre>
<p><strong>Key Finding</strong>: The password is stored in plaintext in the binary. This is a common vulnerability in amateur code.</p>
<h2>Disassembly with GDB</h2>
<p>Let's use GDB to understand the assembly:</p>
<pre><code class="language-bash">gdb ./verify_bin
(gdb) disassemble main
</code></pre>
<p>Looking at the <code>main</code> function, we can see:</p>
<ol>
<li>Stack space is allocated for the buffer</li>
<li><code>printf</code> is called to display the prompt</li>
<li><code>scanf</code> reads user input</li>
<li>The <code>verify_password</code> function is called</li>
<li>Conditional jump based on the return value</li>
</ol>
<h2>Analysis of verify_password</h2>
<p>The <code>verify_password</code> function contains:</p>
<ul>
<li>A stack-allocated string with the password</li>
<li>A call to <code>strcmp</code> to compare input with the hardcoded password</li>
<li>Returns 1 on success, 0 on failure</li>
</ul>
<h2>Security Implications</h2>
<p>This binary demonstrates several security issues:</p>
<ol>
<li><strong>Plaintext Password Storage</strong>: The password is hardcoded and visible via <code>strings</code></li>
<li><strong>No Input Validation</strong>: The scanf could be exploited with buffer overflow</li>
<li><strong>Timing Attacks</strong>: The string comparison could leak information about correct characters</li>
<li><strong>No Authentication Mechanism</strong>: Only a single string comparison for access</li>
</ol>
<h2>Proper Implementation</h2>
<p>Here's how this should be done in a real application:</p>
<pre><code class="language-c">#include &#x3C;stdio.h>
#include &#x3C;string.h>
#include &#x3C;crypt.h>

int verify_password(const char *input, const char *hash) {
    // Use a proper hashing algorithm like bcrypt or Argon2
    // Never store plaintext passwords
    struct crypt_data data;
    char *encrypted = crypt_r(input, hash, &#x26;data);
    return (encrypted != NULL &#x26;&#x26; strcmp(encrypted, hash) == 0);
}
</code></pre>
<h2>Key Takeaways</h2>
<ul>
<li>Always examine binaries with <code>strings</code>, <code>file</code>, and <code>readelf</code></li>
<li>Use GDB and radare2 for deeper analysis</li>
<li>Never store credentials in plaintext</li>
<li>Use proper cryptographic hashing for passwords</li>
<li>Validate all input thoroughly</li>
</ul>
